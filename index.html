OPERATING-SYSTEM ARCHITECTURE

Fenrua in the AI stack

Fenrua does not replace applications, agents, models, tools, infrastructure, or chains. It sits beneath them as the security substrate that constrains execution, records evidence, and enables independent review.

USERS, OPERATORS, ORGANISATIONS AI APPLICATIONS AND WORKFLOWS AGENTS, MODELS, TOOLS, ORCHESTRATORS COMPUTE, DATA, APIS, CLOUD, CHAINS FENRUA USER-SPACE INTEGRATIONS FENRUA SECURITY KERNEL IDENTITY · AUTHORITY · INTEGRITY · POLICY · EVIDENCE · VERIFICATION · CONTAINMENT · RECOVERY
01

Pre-execution policy evaluation

Entity, operator, tool, environment, and requested action are checked against signed authority policy before execution is allowed.

02

Tool-call authorisation

Allowed, denied, and human-approval actions are explicit. Scopes cover files, networks, repositories, deployments, wallets, and data.

03

Runtime-integrity verification

Reviewed source, build digest, dependency lock, container image, model identity, prompt policy, and deployment manifest are compared.

04

Evidence-bundle creation

Inputs, outputs, policy decisions, approvals, hashes, signatures, commands, findings, and limitations become machine-readable records.

05

Revocation and quarantine

Missing identity, stale evidence, drift, invalid signatures, revoked artifacts, and policy violations fail closed and preserve incident evidence.

06

Research-to-kernel translation

Observations become claims, non-claims, threats, vectors, implementations, validation, verifiers, evidence, utilities, and regressions.

KERNEL SPACE

Eight canonical primitives

Kernel-space functions are foundational security primitives. They are not product promises by themselves; each primitive carries its own maturity label, evidence path, and non-claim boundary.

Implemented / reference mix

Identity

AI entities, artifacts, models, tools, builds, operators, deployments, and evidence bundles.

Specification / prototype

Authority

Permitted actions, prohibited actions, delegation, expiry, revocation, scopes, and human approval.

Reference implementation

Integrity

Source commits, build digests, dependency locks, runtime images, policies, contracts, and environment manifests.

Specification

Policy

Filesystem, network, tool, repository, secret, database, signing, deployment, and failure-mode controls.

Implemented / expanding

Evidence

Inputs, outputs, tool calls, policy decisions, builds, tests, findings, deployments, incidents, and recovery events.

Reference implementation

Verification

Signatures, hashes, manifests, completeness, conformance, runtime state, audit scope, and release lineage.

Specification

Containment

Fail-closed responses for stale evidence, drift, invalid identity, invalid signature, missing approval, and revoked artifacts.

Planned / doctrine

Recovery

Revocation, quarantine, key rotation, policy replacement, rollback, evidence continuity, and controlled re-entry.

Contract Evidence Freeze

Contract evidence refresh in progress. Exact source, invariant, deployment, bytecode, and runtime details will be published from the next frozen contract evidence bundle.
